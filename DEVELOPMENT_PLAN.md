# 阳光小镇物语：独立游戏开发进度方案

本文档用于把当前可玩原型推进到“优秀独立游戏”水准。它不是单纯功能清单，而是围绕核心乐趣、制作节奏、质量门槛和发布目标制定的阶段计划。

## 1. 当前项目判断

### 1.1 现有基础

当前项目已经不是静态展示，而是具备完整纵切面的 3D 休闲城市经营原型：

- 18 x 18 固定网格地图。
- Three.js 斜俯视 3D 小镇场景。
- 普通道路和樱花大道两级道路。
- 道路自动连接，支持直线、转角、T 字和十字路口视觉。
- 住宅、商业、工业、公园、学校、消防、电力、水塔、拆除工具。
- 居民对象、住宅到目的地的 A* 道路寻路、通勤路线缓存。
- 道路流量、容量、拥堵率、交通评分。
- 最多 60 个基于真实居民路线采样的小车/行人动画。
- 周结算、收入、维护、人口、就业、幸福度、基础设施覆盖。
- 城市顾问面板能反馈断路、不可达、拥堵、缺电、缺水、就业不足等问题。
- Playwright 冒烟测试覆盖核心玩法。

这意味着项目当前处于“可玩原型 / vertical slice 雏形”阶段，下一步不应盲目加内容，而应先把核心体验打磨成稳定、可重复、可理解、可扩展的游戏循环。

### 1.2 主要优势

- 题材方向清晰：日系、阳光、休闲、小镇建设，与深色赛博和硬核调度形成明显区分。
- 技术栈轻量：原生 JS + Three.js + Python 静态服务，适合快速迭代和控制复杂度。
- 核心系统已经互相连接：道路影响通勤，通勤影响就业和税收，服务覆盖影响幸福度。
- 测试接口已经存在，后续可以持续扩大自动化测试。
- 中文界面统一，当前产品气质比较明确。

### 1.3 主要短板

- 没有存档/读档，刷新即丢失进度，无法形成长期经营体验。
- 缺少明确的阶段目标、任务链和内容节奏，玩家容易不知道下一步追求什么。
- 数值模型可运行，但还缺少可解释性、长期压力曲线和失败/恢复空间。
- 建筑和道路只有基础等级，缺少“成长感”和中后期选择。
- UI 能显示信息，但缺少更细的原因拆解、趋势反馈和操作效率。
- 音频、动效、反馈、镜头、教程等独立游戏质感层还没有系统建设。
- 当前开发环境绑定本机绝对路径，`node_modules` 不在仓库，跨环境复现不够稳。
- 测试目前偏冒烟，缺少存档、长期模拟、边界条件、性能和视觉回归验证。

### 1.4 当前风险

- 继续横向加建筑会让系统变大，但不一定更好玩。
- 如果没有存档和目标章节，玩家很难产生“我要再玩一会儿”的动力。
- 单文件 `src/app.js` 已超过 1000 行，后续扩展会越来越难维护。
- 机器绑定的 Python/npm 路径会影响协作、测试和发布。
- 画面全由基础几何体生成，风格可以成立，但需要更强的视觉层次、地标和动画细节来避免廉价感。

## 2. 产品定位

### 2.1 一句话定位

一款明亮、可爱、轻策略的 3D 小镇建设游戏：玩家通过规划道路、服务设施和生活街区，让居民在樱花大道之间顺畅生活，逐步把小镇发展成有记忆点的晴日港。

### 2.2 核心体验支柱

- 温暖感：画面、文案、反馈和节奏都应让玩家觉得轻松、明亮、治愈。
- 小决策：每次扩建都应有清晰取舍，例如道路容量、维护成本、污染距离、服务覆盖。
- 可读性：玩家必须能快速知道问题在哪里、为什么发生、怎样改善。
- 成长感：从几栋房子到有街区、有地标、有节日的小镇，阶段变化要明显。
- 低压力但不空洞：不追求高压失败，但要有拥堵、预算、服务不足等可恢复挑战。
- 可信模拟：居民、道路、就业和服务不需要真实城市级复杂，但必须让玩家感觉规则一致。

### 2.3 目标玩家

- 喜欢城市建设、模拟经营、放置规划、治愈系小品游戏的玩家。
- 想要比纯装饰游戏更有规则，但不想被硬核资源链压迫的玩家。
- 桌面 16:9 游玩为主，面向键鼠和大屏。

### 2.4 成功标准

到 1.0 版本，项目至少应达到：

- 新玩家 3 分钟内能理解如何开始建设。
- 15 分钟内能完成第一个目标章节。
- 45 到 90 分钟有明显的中期规划压力和扩建动力。
- 一局至少支持 2 到 4 小时轻量经营。
- 自动化测试覆盖核心规则、存档、关键 UI、长期模拟和发布构建。
- 画面、音效、反馈和文案形成统一的“阳光小镇”识别度。

## 3. 版本路线总览

推荐采用 6 个阶段推进：

| 阶段 | 目标 | 建议周期 | 版本目标 |
| --- | --- | --- | --- |
| P0 | 稳定开发环境和代码地基 | 1 周 | `0.1 稳定原型` |
| P1 | 补齐核心游戏闭环 | 2-3 周 | `0.2 可保存的核心循环` |
| P2 | 建立进度、任务和成长系统 | 3-4 周 | `0.3 章节化试玩版` |
| P3 | 增加内容深度和城市个性 | 4-6 周 | `0.5 内容 Alpha` |
| P4 | 体验打磨、音画表现和性能 | 3-5 周 | `0.7 Demo / Beta` |
| P5 | 发布准备、QA、平衡和打包 | 3-4 周 | `1.0 发布候选` |

整体节奏按单人或小团队估算约 16 到 23 周。如果只做公开 Demo，P0 到 P4 的前半段即可，目标约 10 到 14 周。

## 4. P0：稳定开发环境和代码地基

状态：已完成。项目已新增 `environment.yml`、`scripts/setup-dev.bat`、`scripts/setup-dev.ps1`、`tools/env.js`、`tools/serve.js`、`tools/check-env.js` 和 `.env.example`，`package.json`、测试 runner、后台启动器、Playwright 配置和双击启动脚本已改为使用统一环境解析。本机已创建 `sunny-town-dev` conda 环境，安装 npm 依赖和 Playwright Chromium，并验证 `npm.cmd test` 通过 7 项冒烟测试。

### 目标

让项目在新机器上可以稳定安装、运行、测试，并为后续扩展降低维护压力。

### 开发任务

- 移除或封装硬编码 Python 绝对路径。
- 提供跨环境启动策略，例如优先使用 `py`/`python`，必要时允许 `.env` 指定 Python。
- 增加 `setup`/`check` 类脚本，检查 Node、npm、Python、Playwright 浏览器是否可用。
- 把 README 和 HANDOFF 中旧路径同步为当前项目路径或改为通用说明。
- 梳理 `src/app.js`，先不大改架构，但标记后续拆分边界：
  - constants/config
  - city state/model
  - simulation
  - pathfinding/traffic
  - rendering
  - ui
  - test api
- 增加基础错误兜底：依赖缺失、服务启动失败、端口占用时给出可读提示。
- 确认 `.gitignore` 覆盖本地日志、截图、报告和临时状态。

### 验收标准

- 新环境执行安装命令后可以启动本地服务。
- 测试命令能一键启动服务并运行 Playwright。
- README 的运行说明不依赖某台机器的绝对路径。
- 当前 7 个冒烟测试全部通过。

### 已完成项

- `package.json` 不再写死旧 Anaconda Python 路径。
- 新增 `environment.yml`，定义 `sunny-town-dev` 的 Python/Node 开发环境。
- 新增 `scripts/setup-dev.bat` 和 `scripts/setup-dev.ps1`，支持一键创建/更新 conda 环境、安装 npm 依赖和 Playwright 浏览器。
- `tools/run-tests.js` 和 `tools/start-server.js` 通过 `tools/env.js` 解析 Python、host 和 port。

- 新增 `.env.example`，支持用 `SUNNY_TOWN_PYTHON`、`SUNNY_TOWN_HOST`、`SUNNY_TOWN_PORT` 覆盖本地设置。
- `start-sunny-town.bat` 支持 `.env`、`py -3`、`python`，并可自动复用 `sunny-town-dev`。
- README 和 HANDOFF 已同步新的通用环境说明。

### 验证结果

- `sunny-town-dev`：Python 3.12.13、Node.js 22.22.3、npm 10.9.8。

## 5. P1：补齐核心游戏闭环

### 目标

让玩家能保存进度、理解问题、持续扩建，并让一局游戏有最基本的开始、中段和达成。

### 核心功能

- `localStorage` 存档/读档。
- 新游戏、继续游戏、重置城镇。
- 自动存档和手动存档。
- 存档版本号和迁移策略，避免后续结构变化直接坏档。
- 道路升级工具：普通道路可以直接付费升级为樱花大道。
- 更详细的地块详情：
  - 建筑维护费、税收、服务类型。
  - 电力、水力、教育、消防、公园覆盖。
  - 是否道路可达。
  - 关联居民数量、平均通勤。
- 更清晰的顾问建议：
  - 问题。
  - 原因。
  - 推荐行动。
- 游戏内基础教程：
  - 第一步铺路。
  - 第二步建住宅。
  - 第三步建商业/工业。
  - 第四步补电力/水力。
  - 第五步观察通勤和拥堵。

### 数值任务

- 明确人口增长公式和服务影响。
- 明确收入与维护的早期平衡。
- 确保前 10 分钟不轻易破产，也不会无脑滚雪球。
- 增加低资金恢复空间，例如税收等待、拆除返还、低维护规划。

### 验收标准

- 玩家刷新页面后能继续同一座小镇。
- 玩家能把普通道路升级为樱花大道，并看到拥堵改善。
- 顾问面板对至少 6 类常见问题给出可执行建议。
- 新玩家不看 README 也能完成第一个目标。
- 自动化测试覆盖存档、读档、道路升级和重置。

## 6. P2：建立进度、任务和成长系统

### 目标

从“能玩”变成“想继续玩”。玩家需要阶段目标、奖励、解锁和小镇成长叙事。

### 章节设计

建议第一版做 5 个章节：

| 章节 | 名称 | 目标 | 解锁/奖励 |
| --- | --- | --- | --- |
| 1 | 新居民入住 | 人口 80，幸福度 60%，道路不断连 | 公园、周报 |
| 2 | 商店街开张 | 就业率 70%，商业 2 栋，交通 75% | 樱花大道升级折扣 |
| 3 | 安心生活圈 | 电力/水力 85%，消防 40%，教育 30% | 学校/消防提示强化 |
| 4 | 春日通勤网 | 人口 450，交通 72%，平均拥堵受控 | 小车/行人外观变化 |
| 5 | 阳光小镇成型 | 人口 800，幸福度 75%，资金为正 | 结算庆祝、自由模式 |

### 系统任务

- 章节目标面板。
- 目标完成动画和奖励反馈。
- 周报系统：人口、资金、交通、幸福度趋势。
- 轻量成就：
  - 第一次无拥堵周。
  - 第一条樱花大道。
  - 第一片完整生活圈。
  - 连续 10 周资金为正。
- 建筑升级雏形：
  - 住宅 -> 小公寓。
  - 商业 -> 商店街。
  - 工业 -> 工坊。
- 升级条件应依赖服务覆盖、道路可达和资金，不只是点击付钱。

### 内容任务

- 增加 3 到 5 个视觉地标或装饰：
  - 便利店。
  - 小车站。
  - 神社或小广场。
  - 小桥或水渠。
  - 祭典灯笼。
- 这些内容可以先作为功能建筑或装饰奖励，不必一次做复杂规则。

### 验收标准

- 玩家有清晰的 5 段成长路径。
- 每个章节至少引导一个核心系统。
- 完成章节时有明显视觉/文案反馈。
- 一局能稳定推进到 800 人目标。
- 自动化测试覆盖章节达成和基础升级。

## 7. P3：增加内容深度和城市个性

### 目标

让小镇规划有更多有效选择，并让每局城市看起来不完全一样。

### 玩法扩展

- 分区倾向：
  - 住宅喜欢公园、学校、低污染、低拥堵。
  - 商业喜欢居民密度和道路通达。
  - 工业喜欢道路容量，讨厌离住宅太近。
- 服务设施容量：
  - 学校、消防、电力、水塔不只是半径，也有容量上限。
- 道路层级策略：
  - 普通道路适合支路。
  - 樱花大道适合主干。
  - 后期可考虑“步行街”或“公交站”，但要谨慎，避免系统失控。
- 事件系统：
  - 春日祭典：短期提升幸福和交通压力。
  - 新店开张：商业税收提升。
  - 暴雨周：道路效率下降，水力设施价值提升。
  - 搬家潮：幸福度低时人口流失。
- 需求/满意度原因拆分：
  - 通勤。
  - 服务。
  - 环境。
  - 生活便利。
  - 财务安全。

### 视觉扩展

- 建筑升级后外观明显变化。
- 道路拥堵视觉更细：
  - 轻微拥堵。
  - 严重拥堵。
  - 断路/不可达提示。
- 更多居民反馈气泡，但要控制频率。
- 小镇地标和节日装饰形成阶段变化。

### 技术扩展

- 将核心系统逐步拆分为模块。
- 建立可序列化的城市数据模型，渲染对象不进入存档。
- 增加长期模拟测试，例如自动推进 200 周，验证没有 NaN、负容量、坏路径和性能异常。
- 增加性能基线：
  - 1440 x 900 稳定 50-60 FPS。
  - 18 x 18 满图情况下交互不卡顿。
  - 移动体上限和粒子上限可配置。

### 验收标准

- 中期规划至少有 3 种有效路线：服务优先、商业优先、交通优先。
- 建筑升级改变数值和外观。
- 事件不会随机毁掉玩家，而是制造短期调整目标。
- 长期模拟测试通过。

## 8. P4：体验打磨、音画表现和性能

### 目标

把“原型味”降下来，把独立游戏的质感补上。

### UI/UX 打磨

- 统一按钮状态、禁用原因、悬停提示。
- 增加撤销最近一次建造，至少支持 1 步撤销。
- 支持快捷键：
  - 道路。
  - 住宅。
  - 商业。
  - 拆除。
  - 暂停/速度。
  - 取消选择。
- 增加镜头回中、缩放边界、拖拽手感调整。
- 减少 HUD 遮挡，确保 16:9 下信息密度舒服。
- 让所有失败操作都有明确原因，而不是只“不响应”。

### 视觉打磨

- 日夜或季节轻变化，优先做季节，不必做真实昼夜。
- 云、樱花、树、车流、建筑落地动画统一节奏。
- 优化材质层次，避免所有物体像同一亮度的塑料块。
- 给关键建筑做更强轮廓：
  - 学校。
  - 消防站。
  - 水塔。
  - 商店街。
  - 小车站/神社。
- 完成目标时有庆祝镜头和小镇活动。

### 音频

- 背景音乐开关。
- UI 点击音。
- 建造音。
- 拆除音。
- 金币/人口增长音。
- 章节完成音。
- 拥堵或问题提示音要克制，不能破坏休闲感。

### 性能和兼容

- 只保留桌面 16:9 为主，但要明确最低窗口尺寸。
- Canvas 尺寸、像素比和阴影质量可控。
- 避免每帧创建大量对象。
- 检查长时间运行内存是否持续增长。

### 验收标准

- 完整游玩 60 分钟无明显卡顿、报错或 UI 错乱。
- 玩家能通过界面理解多数规则。
- 音画反馈显著提升操作手感。
- Playwright 视觉截图稳定生成。

## 9. P5：发布准备、QA、平衡和打包

### 目标

让项目具备公开发布或提交 Demo 的质量。

### 发布内容

- 完整 README：
  - 游戏介绍。
  - 运行方式。
  - 操作说明。
  - 测试方式。
  - 开发路线。
- 游戏内帮助页。
- 版本号和更新日志。
- 截图和封面图。
- 可离线运行的打包方案：
  - 简单版：静态网页包。
  - 桌面版：后续可考虑 Electron/Tauri，但不建议过早引入。
- 许可证和第三方依赖说明。

### QA 清单

- 新游戏。
- 继续游戏。
- 重置存档。
- 存档迁移。
- 全部建筑建造/拆除。
- 道路升级/拆除。
- 章节完成。
- 资金为负。
- 无道路、断路、满图。
- 长期运行。
- 浏览器刷新。
- 不同窗口尺寸下 UI 不重叠。

### 平衡目标

- 早期：3 到 8 分钟学会。
- 中期：15 到 45 分钟出现规划压力。
- 后期：45 到 120 分钟仍有优化目标。
- 失败：允许资金紧张，但不要一错即死。
- 成功：目标达成后仍可自由扩建。

### 验收标准

- 关键测试全部通过。
- 没有已知阻断级 bug。
- 首次游玩体验可完整闭环。
- 发布包和源码仓库都可复现运行。

## 13. 质量门槛

每次合入重要功能前，至少满足：

- `npm.cmd test` 通过。
- 新功能有至少 1 个自动化测试或清晰手动验证说明。
- 不破坏中文界面和桌面 16:9 布局。
- 不提交 `node_modules`、测试截图、报告、日志、pid 文件。
- 存档结构变更必须考虑版本迁移。
- 新增规则必须能被 UI 或顾问解释。

## 14. 第一阶段推荐执行顺序

当前最推荐先做：

1. 修复开发环境可复现性。
2. 实现存档/读档。
3. 实现道路升级。
4. 增加章节目标。
5. 扩展顾问和地块详情。

原因是这五项能直接把项目从“演示型原型”推向“玩家愿意连续游玩”的方向，并且不会过早制造无法维护的内容负担。
